# 知识文档模板与裁剪指南

本文用于指导 `docs/knowledge/` 下知识文档的补充和重写。

目标不是让每个模块都写成大而全的长文档。
每篇文档应按风险和复杂度写到足够可用：
能帮后来者快速定位代码、理解关键行为、判断改动影响，并选择合适的验证方式。

## 使用原则

- 先判断文档等级，再决定写作深度。
- 保留模块差异；不适用的章节可以删掉，或用一句话说明“不涉及”。
- 所有路径均以本仓根目录为基准，例如 `rosen/modules/render_service/...`。
- 重要结论要有代码锚点；没有代码依据的经验结论要标明来源或待确认。
- 涉及公开 API、IPC、buffer、显示设备、性能功耗或跨仓接口时，必须写清验证缺口。

## 文档等级

- L1 路由卡片：适用于低风险目录、简单工具、很少改动的模块。
  推荐 50-120 行，包含适用范围、代码地图、核心模型、验证建议。
- L2 标准模块文档：适用于常见开发模块和边界清晰的模块。
  推荐 120-250 行，包含 L1 内容，并补充数据流、修改指南、检查清单。
- L3 深度行为专题：适用于跨进程、跨线程、跨语言或设备相关能力。
  推荐 250 行以上，包含 L2 内容，并补充行为限制、生命周期、失败路径、排障、兼容性。

优先把以下主题写成 L3：

- IPC、Parcel、Command、SAFuzz 和不可信输入。
- SurfaceBuffer、PixelMap、Image、Cache、GPU/HWC 资源生命周期。
- HGM、VSync、刷新率投票、LTPO、真实设备显示效果。
- Drawable、Modifier、Animation 的跨线程同步和生命周期。
- NAPI、NDK、CJ、ANI、Taihe 等公开 API 或错误码行为。
- HDR、颜色空间、HPAE、GL/VK/native buffer 等设备能力相关路径。

## 裁剪规则

写文档时按下面规则裁剪：

- L1 文档可以删除“不适用范围”“生命周期”“线程边界”“常见故障排查”等章节。
- 没有公开接口的模块，可以删除“外部可见行为与兼容性”。
- 没有资源所有权或跨线程逻辑的模块，可以删除“生命周期与所有权”和“线程边界”。
- 设备无关的模块，不要写设备验证结论；只写不需要设备验证的原因。
- 如果某章节暂时没有稳定结论，不要硬编，放到“待补充背景”。

## 写作质量要求

- 少写“可能、应该、通常”这类模糊判断；必要时说明依据。
- 不把当前实现误写成长期规格，尤其是硬编码阈值、产品开关和 vendor 扩展。
- 不把未验证的设备行为、显示效果、性能功耗写成完整结论。
- 不复制大段代码；用符号、文件路径和关键分支说明即可。
- 表格适合列锚点和限制；流程适合用文本箭头或 Mermaid；排障适合用检查清单。

## 评审检查

提交知识文档前至少检查：

- `AGENTS.md` 是否已经把对应场景路由到本文。
- 文档中的仓内路径是否存在，或是否明确标注为外部仓/设备路径。
- 是否写清哪些结论来自代码、哪些来自团队经验、哪些仍待确认。
- 是否列出最近验证目标和当前环境下无法验证的缺口。
- 是否没有旧入口、重复路由或互相矛盾的说明。
